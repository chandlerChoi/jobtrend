잡트렌드 (JobTrend) — 기술명세서 v1.0
1. 프로젝트 개요
한 줄 소개: 지금 어떤 공고가 있는지, 직무별로 어떤 자격요건과 트렌드가 필요한지 보여주고, 원하는 키워드를 등록하면 알람·데일리 리포트로 받아볼 수 있으며, AI 모의면접까지 지원하는 구직자용 채용 인텔리전스 서비스.
해결하려는 문제

정보 비대칭 — 기업은 ATS로 지원자를 정교하게 거르지만, 구직자는 "이 직무에 합격하려면 어떤 자격요건이 필요한지" 사전에 파악할 도구가 거의 없음
기회 누락 — 채용이 줄어든 시장에서 본인 스킬셋이 통하는 인접 직무를 놓치고 한 직무에만 집중하는 경우가 많음
면접 준비 비대칭 — AI 면접질문 추천은 기업용으로는 상용화됐지만, 구직자가 직접 쓸 수 있는 도구는 비어 있음

핵심 가치 제안
영역제공 가치트렌드 파악직무별 자격요건·기술스택 키워드를 실제 공고 데이터로 통계화알림·구독키워드 등록 시 신규 공고 알림 + 매일 트렌드 변화 리포트 발송기회 확장자격요건이 겹치는 인접 직무 자동 추천면접 준비JD+이력서 기반 AI 예상질문 생성, 모의면접·피드백
시장조사 요약: 기업의 65.7~73.4%가 2026년 정규직 채용 계획 있음. 신규채용 시급 직군은 제조·현장·기능직(44.8%), 연구개발(34.2%). 수시채용만 하는 기업 70.8%, 직무중심 채용 강화가 주요 트렌드(72.2%). 중고신입 선호 강화(33.5%, HR이슈 1위), 4~7년차 경력직 선호(49.7%). 평가 시 "직무 관련 업무 경험"이 가장 중요(67.6%).
경쟁 서비스 비교

나인하이어/라운드HR: 기업(채용담당자) 대상, 스크리닝 자동화
합격닷컴/AI ResumeMaker: 개인 대상, 1:1 자소서 첨삭(수동)
잡트렌드(본 서비스): 개인 대상, 트렌드 파악+알림 구독+모의면접 통합, 구독+크레딧 과금

UX 페르소나 3종

김신입(26세, 취준생) — 합격 기준 사전 파악 필요
박전환(31세, 이직준비) — 인접 직무 탐색 필요
이재직(29세, 면접 D-3) — 실전 모의면접 연습 필요


2. 핵심 기능 (4종, 모두 v1.0 MVP 범위)
F1. 채용 트렌드 & 자격요건 분석 대시보드

직무명 입력 → 사람인·잡코리아 API로 공고 수집 → 경력수준 분포, 학력요건 분포, 키워드 Top15, 최근 30일 공고 수 추이 제공. 최우선 기능 (다른 3개의 데이터 기반).
F2. 키워드 알람 & 데일리 리포트 구독

관심 키워드(직무명·기술스택·지역·연차, 무료 최대 5개) 등록 → 신규 공고 알림 + 전일 대비 트렌드 변화 데일리 다이제스트. F1 데이터 재사용.
F3. 인접 직무 추천

관심 직무 입력 → 자격요건 키워드 유사도 상위 3~5개 인접 직무 + 겹치는 핵심 스킬(추천 근거) 제공. F1의 키워드 집계 결과에 코사인 유사도 계산만 추가.
F4. AI 모의면접 시뮬레이터

JD+이력서 입력 → Claude API가 예상 질문 5~7개 생성, 텍스트 모의면접 진행, 답변 피드백. 무료→유료 전환의 핵심 — 무료 크레딧(월 2~3회) 소진 후 충전 유도.
기능 간 관계: F1→F2(신규/변화 감지), F1→F3(유사도 계산 입력), F1→F4(질문 생성 맥락)

3. 화면 구성
화면기능구성요소온보딩/직무 검색공통검색창, 인기 직무 태그트렌드 대시보드F1분포 차트, 워드클라우드, 추이 그래프키워드 알람 설정F2등록 폼, 채널 선택데일리 리포트함F2날짜별 리포트, 변화 하이라이트인접 직무 추천F3추천 카드(유사도%, 스킬 태그)AI 모의면접F4입력 폼, 채팅형 UI, 피드백마이페이지공통키워드 관리, 크레딧 현황, 결제
4. 사용자 흐름
가입 → 직무 검색(F1) → 키워드 등록(F2) → (선택) 인접 직무 확인(F3) → 데일리 리포트 수신 → JD 입력 후 모의면접(F4) → 크레딧 소진 시 결제
전환 설계: F1·F2·F3는 무료 개방(가입·재방문 유도), F4가 실제 매출 발생 지점.

5. 시스템 구조
4개 기능 모두 동일한 4-agent 파이프라인(Collector→Analyzer→Insight→Report) 공유.
계층구성ClientReact+Vite, Vercel 정적 배포API LayerVercel Serverless FunctionsExternal APIs사람인 API, 잡코리아 API, Claude APIDatabaseNeon DB (Serverless PostgreSQL)SchedulerVercel Cron (1일 1회 Collector 배치)
기능별 매핑

F1: 공고 수집 → 필드 집계 → 패턴 도출 → 차트 렌더링
F2: 신규 공고 수집(F1 재사용) → 키워드 매칭 → 변화 감지 → 알림 발송
F3: F1 집계 재사용 → 코사인 유사도 계산 → 상위 N개 선정 → 추천 카드 표시
F4: JD/이력서 입력 → Claude API 질문 생성 → 답변 평가 → 피드백 생성

7. 사용 API

사람인 API: 이용신청 승인 필요, 1일 최대 500회. 직무·지역·경력·학력·연봉코드·키워드 제공
잡코리아 API: 신청서 검토 후 승인, 사용목적 불분명 시 거절 가능(승인 지연 리스크)
Claude API: F4 질문생성·피드백, F1 트렌드 요약 보조

→ 두 채용 API 모두 합격자 개인 데이터는 제공 안 함 — 이미 "공고 자격요건 트렌드"로 범위 조정 반영됨.
AI 에이전트 활용: Claude(기획·BMC·페르소나), Perplexity(트렌드 검색), Claude Code(구현), Claude API(서비스 내재 LLM) — 필수조건(AI 에이전트 2종+, 코딩에이전트 1종+, LLM API 1종+, Open API 1종+) 모두 충족.

4. DB 스키마 (Neon DB, 실제 DDL)
sqlCREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255),
  plan_tier VARCHAR(20) NOT NULL DEFAULT 'free',
  interview_credits INT NOT NULL DEFAULT 3,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE job_postings (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  source VARCHAR(20) NOT NULL,              -- 'saramin' | 'jobkorea'
  external_id VARCHAR(100) NOT NULL,
  title VARCHAR(255) NOT NULL,
  company VARCHAR(255),
  job_category VARCHAR(100) NOT NULL,
  region VARCHAR(100),
  employment_type VARCHAR(50),
  experience_min INT,
  experience_max INT,
  education_level VARCHAR(50),
  salary_code VARCHAR(50),
  keywords TEXT[],
  raw_requirements TEXT,
  posting_url TEXT,
  posted_at DATE,
  collected_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(source, external_id)
);
CREATE INDEX idx_postings_category ON job_postings(job_category);
CREATE INDEX idx_postings_collected ON job_postings(collected_at);

CREATE TABLE job_category_stats (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  job_category VARCHAR(100) NOT NULL,
  keyword VARCHAR(100) NOT NULL,
  frequency INT NOT NULL DEFAULT 0,
  period_date DATE NOT NULL,
  UNIQUE(job_category, keyword, period_date)
);

5. 핵심 알고리즘 로직 (의사코드 수준)
F1 — 키워드 추출

raw_requirements 텍스트를 사전 정의 기술스택 딕셔너리(Python/Java/AWS/React/SQL 등 약 150~200개)와 매칭
경력수준은 experience_min/max를 4구간(신입/1-3년/3-5년/5년+)으로 버킷화
매일 배치로 job_category_stats에 (job_category, keyword, frequency, period_date=오늘) UPSERT
percentage = frequency / 해당일 총 공고수 * 100
Top 15는 frequency DESC LIMIT 15

F2 — 변화 감지

Cron이 오늘자 통계를 만든 뒤 어제자와 비교: delta% = (today.freq - yesterday.freq) / yesterday.freq * 100
|delta%| >= 20이면 frequency_spike/frequency_drop으로 하이라이트
사용자 keyword_alerts와 매칭되는 collected_at = 오늘인 신규 공고는 new_posting으로 별도 포함
결과를 daily_reports.content_json에 저장 후 이메일 발송

F3 — 인접 직무 추천 (코사인 유사도)

직무별 키워드 벡터 구성: vector[keyword] = frequency (최신 period_date 기준)
두 직무 A, B의 유사도 = (A·B) / (|A| × |B|)
전체 직무 쌍에 대해 주 1회 배치 계산 → job_similarity 갱신
조회 시 WHERE job_category_a = A ORDER BY similarity_score DESC LIMIT 5
shared_keywords는 두 벡터 모두 frequency>0인 키워드 상위 5개

function extractIdFromUrl(url) {

  return m ? m[1] : url;
}

정규화 코드 (XML 파서 포함, 태그명만 나중에 교체)
javascript// api/lib/normalizers.js (이어서)
const { XMLParser } = require('fast-xml-parser'); // npm install fast-xml-parser

function normalizeJobKoreaJob(rawJob) {
  const careerText = rawJob.gi_career_nm ?? '';
  const { min, max } = parseExperienceRange(careerText);
  return {
    source: 'jobkorea',
    external_id: String(rawJob.gi_no),
    title: rawJob.gi_title,
    company: rawJob.co_name,
    job_category: mapJobKoreaCategory(rawJob.gi_job_cd_nm),  // 3번 매핑 테이블 통과
    region: rawJob.gi_area_nm,
    employment_type: null,        // 잡코리아 필드 확인 후 채움
    experience_min: min,
    experience_max: max,
    education_level: rawJob.gi_edu_nm,
    salary_code: null,            // 잡코리아는 코드 대신 텍스트(gi_salary_nm)일 가능성 → 별도 컬럼 고려
    keywords: extractKeywordsFromText(rawJob.gi_duty),  // 자연어 → 키워드 추출(5장 알고리즘)
    posting_url: rawJob.gi_url,
    raw_requirements: rawJob.gi_duty,
  };
}

function parseExperienceRange(text) {
  if (text.includes('신입')) return { min: 0, max: 0 };
  const m = text.match(/(\d+)\s*~\s*(\d+)/);
  if (m) return { min: Number(m[1]), max: Number(m[2]) };
  const single = text.match(/(\d+)\s*년\s*이상/);
  if (single) return { min: Number(single[1]), max: null };
  return { min: null, max: null };
}

async function fetchJobKoreaJobs(jobCode, page = 1) {
  const url = `http://openapi.jobkorea.co.kr/...?key=${process.env.JOBKOREA_API_KEY}&jobCode=${jobCode}&page=${page}`;
  const res = await fetch(url);
  const xmlText = await res.text();
  const parsed = new XMLParser().parse(xmlText);
  const jobs = parsed.jobs.job; // 실제 루트 태그명은 승인 후 확인
  return (Array.isArray(jobs) ? jobs : [jobs]).map(normalizeJobKoreaJob);
}

module.exports = { normalizeJobKoreaJob, parseExperienceRange, fetchJobKoreaJobs };

3. 직무 카테고리 통합 매핑 (두 API의 직무명이 다른 문제 해결)
사람인은 job-mid-code(예: "개발·데이터"), 잡코리아는 gi_job_cd_nm(예: "IT개발" 등 다른 명칭)을 쓸 가능성이 높아서, 두 소스를 같은 job_category로 묶는 통합 매핑 테이블이 반드시 필요해요. 이게 없으면 F1 대시보드에서 같은 직무인데 두 개로 쪼개져서 집계됨.
javascript// api/lib/categoryMap.js
// 내부 표준 카테고리명을 기준으로 양쪽 소스 원본명을 매핑
const CATEGORY_MAP = {
  "백엔드 개발자": { saramin: ["개발·데이터"], jobkorea: ["IT개발", "백엔드"] },
  "프론트엔드 개발자": { saramin: ["개발·데이터"], jobkorea: ["IT개발", "프론트엔드"] },
  "마케팅": { saramin: ["마케팅·홍보·조사"], jobkorea: ["마케팅"] },
  // 실제 코드표 확인 후(사람인 "직무/직업 코드표" 페이지) 전체 채워야 함
};

function mapJobKoreaCategory(rawName) {
  for (const [standard, sources] of Object.entries(CATEGORY_MAP)) {
    if (sources.jobkorea.includes(rawName)) return standard;
  }
  return rawName; // 매핑 안 되면 원본 그대로 (나중에 수동 보정)
}
module.exports = { CATEGORY_MAP, mapJobKoreaCategory };
